title: "Docker deployment"

The official `dorylab/dory` image is the recommended way to self-host Dory. It sets `DORY_RUNTIME=docker` automatically, runs migrations on startup, and exposes port `3000`.

## Prerequisites

- Docker 20.10 or later installed and running
- `openssl` available on your host (used to generate secrets)

## docker run

<Steps>
  <Step title="Generate secrets">
    Generate the two required secret values before running the container:

    ```bash
    echo "DS_SECRET_KEY:      $(openssl rand -base64 32 | tr -d '\n')"
    echo "BETTER_AUTH_SECRET: $(openssl rand -hex 32)"
    ```

    Copy both values — you will use them in the next step.
  </Step>
  <Step title="Run the container">
    ```bash
    docker run -d --name dory \
      -p 3000:3000 \
      -v /data/dory:/data/dory \
      -e DS_SECRET_KEY="<your-ds-secret-key>" \
      -e BETTER_AUTH_SECRET="<your-better-auth-secret>" \
      -e BETTER_AUTH_URL="http://localhost:3000" \
      -e TRUSTED_ORIGINS="http://localhost:3000" \
      -e DORY_AI_PROVIDER=openai \
      -e DORY_AI_MODEL=gpt-4o-mini \
      -e DORY_AI_API_KEY=your_api_key_here \
      -e DORY_AI_URL=https://api.openai.com/v1 \
      dorylab/dory:latest
    ```

    Replace `<your-ds-secret-key>` and `<your-better-auth-secret>` with the values generated in step 1.
  </Step>
  <Step title="Open Dory">
    Navigate to `http://localhost:3000`. The bootstrap script runs migrations automatically before the server starts, so the first startup may take a few extra seconds.
  </Step>
</Steps>

<Note>
  The `-v /data/dory:/data/dory` flag mounts a host directory for PGLite persistence. Without it, all application data is lost when the container is removed. See [Storage configuration](/self-hosting/storage) for details.
</Note>

## Docker Compose

For a more manageable setup, use Docker Compose:

```yaml docker-compose.yml
services:
  dory:
    image: dorylab/dory:latest
    container_name: dory
    restart: unless-stopped
    ports:
      - "3000:3000"
    volumes:
      - /data/dory:/data/dory
    environment:
      DS_SECRET_KEY: "${DS_SECRET_KEY}"
      BETTER_AUTH_SECRET: "${BETTER_AUTH_SECRET}"
      BETTER_AUTH_URL: "${BETTER_AUTH_URL:-http://localhost:3000}"
      TRUSTED_ORIGINS: "${TRUSTED_ORIGINS:-http://localhost:3000}"
      DORY_AI_PROVIDER: "${DORY_AI_PROVIDER:-openai}"
      DORY_AI_MODEL: "${DORY_AI_MODEL:-gpt-4o-mini}"
      DORY_AI_API_KEY: "${DORY_AI_API_KEY}"
      DORY_AI_URL: "${DORY_AI_URL:-https://api.openai.com/v1}"
```

Create a `.env` file alongside `docker-compose.yml`:

```bash .env
DS_SECRET_KEY=<output of: openssl rand -base64 32 | tr -d '\n'>
BETTER_AUTH_SECRET=<output of: openssl rand -hex 32>
BETTER_AUTH_URL=http://localhost:3000
TRUSTED_ORIGINS=http://localhost:3000
DORY_AI_PROVIDER=openai
DORY_AI_MODEL=gpt-4o-mini
DORY_AI_API_KEY=your_api_key_here
DORY_AI_URL=https://api.openai.com/v1
```

Then start Dory:

```bash
docker compose up -d
```

## Volume mounting

Dory's PGLite database stores all application data — users, sessions, saved queries, and connection metadata — at the path configured by `PGLITE_DB_PATH` (default: `/data/dory`).

Always mount this path to a host directory:

```bash
-v /data/dory:/data/dory
```

Or in Docker Compose using a named volume:

```yaml docker-compose.yml
services:
  dory:
    volumes:
      - dory_data:/data/dory

volumes:
  dory_data:
```

<Warning>
  Removing the container without a volume mount destroys all stored data including user accounts, connection configurations, and saved queries.
</Warning>

## Port mapping

The container listens on port `3000`. Map it to any host port:

```bash
-p 3000:3000   # host port 3000 → container port 3000
-p 8080:3000   # host port 8080 → container port 3000
```

If you change the host port, update `BETTER_AUTH_URL` and `TRUSTED_ORIGINS` to match.

## Verifying the container

Check that the container is running:

```bash
docker ps --filter name=dory
```

Expected output:

```
CONTAINER ID   IMAGE                  COMMAND                  STATUS
abc123def456   dorylab/dory:latest    "sh -c 'node dist-sc…"   Up 30 seconds
```

Fetch the health endpoint:

```bash
curl -sf http://localhost:3000/api/health && echo "OK"
```

## Viewing logs

```bash
# Follow live logs
docker logs -f dory

# Last 100 lines
docker logs --tail 100 dory
```

Look for the bootstrap migration output and the Next.js server startup message to confirm a clean start.

## Updating to a new version

<Steps>
  <Step title="Pull the latest image">
    ```bash
    docker pull dorylab/dory:latest
    ```
  </Step>
  <Step title="Stop and remove the current container">
    ```bash
    docker stop dory && docker rm dory
    ```

    Your data is safe as long as the volume (`/data/dory`) is mounted on the host or uses a named volume.
  </Step>
  <Step title="Start the new container">
    Run the same `docker run` command as before (or `docker compose up -d` if using Compose). The bootstrap script runs migrations automatically on startup.
  </Step>
</Steps>

<Tip>
  Pin to a specific version tag (e.g., `dorylab/dory:0.5.0`) in production so that updates are explicit rather than automatic.
</Tip>
